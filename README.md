# SAFE-HOME

AFP(아파트 화재 안전 플랫폼) 데이터를 기반으로, 화재 발생 시 **입주민 · 119 종합상황실 · 현장 소방대원** 3자가
같은 정보를 실시간으로 공유하며 각자 최적의 행동을 결정하도록 돕는 의사결정 플랫폼 데모입니다.

빌드 도구 없이 순수 HTML/CSS/JavaScript로 작성되어 있어, 저장소를 그대로 GitHub Pages에 올리면 바로 동작합니다.

## 실행 방법

### 로컬에서 열어보기
`index.html`을 더블클릭해서 브라우저로 바로 열어도 기본 동작은 확인할 수 있습니다.
다만 **세대 간 실시간 동기화(localStorage 이벤트)** 는 브라우저가 파일을 `file://`로 열 때 탭마다
출처(origin)를 다르게 취급할 수 있어 불안정합니다. 정확히 확인하려면 간단한 정적 서버로 열어주세요.

```bash
# Node.js가 있다면
npx serve .

# Python이 있다면
python -m http.server 8080
```

그 다음 브라우저에서 같은 서버 주소로 탭을 2~3개 열어(예: 입주민 탭 / 119 상황실 탭) 입주민 화면에서
대피 안내를 진행하면, 다른 탭의 상황실·소방대원 화면에 실시간으로 반영되는 것을 확인할 수 있습니다.
(한 탭 안에서 상단 역할 탭을 눌러 전환해도 동일하게 반영됩니다.)

### GitHub Pages로 배포하기
1. 이 폴더 전체를 GitHub 저장소에 push 합니다.
2. 저장소 **Settings → Pages** 에서 Source를 `GitHub Actions`로 선택합니다.
   (이미 `.github/workflows/pages.yml`이 포함되어 있어 `main` 브랜치에 push할 때마다 자동 배포됩니다.)
3. 배포가 끝나면 `https://<계정>.github.io/<저장소명>/` 주소로 접속할 수 있습니다.

Actions 없이도, Settings → Pages에서 `Deploy from branch: main / (root)`를 선택하면 별도 빌드 없이 바로 서빙됩니다.

## 폴더 구조

```
safehome-app/
├─ index.html            앱 셸(상단바, 역할 탭, 뷰 컨테이너)
├─ manifest.webmanifest  PWA 매니페스트 (홈 화면에 설치 가능)
├─ sw.js                 오프라인 대비 앱 셸 캐시 서비스워커
├─ css/styles.css        전체 스타일
├─ icons/icon.svg         앱 아이콘
└─ js/
   ├─ data.js       질문 정의, 등록 건물 디렉터리(BUILDINGS)의 AFP-Core/AFP-Search 목업 DB, 세대 그리드 생성
   ├─ ui.js         공용 DOM 유틸(아이콘 SVG, 토스트, 세대 현황 그리드 렌더러)
   ├─ store.js      3개 화면이 공유하는 상태 저장소 (localStorage + storage 이벤트 + BroadcastChannel)
   ├─ rules.js      Q1~Q5 답변과 AFP 정보로 대피 행동(A~G)을 판정하는 규칙 엔진
   ├─ resident.js   입주민 대피 안내 화면
   ├─ situation.js  119 종합상황실 대시보드 (신고 큐 + 세대 체크리스트, 읽기 전용 모니터링)
   ├─ firefighter.js 현장 소방대원 정보카드
   ├─ admin.js      관리자 모드 (사건 생성/위치 변경, 배포 링크 발급·TTL 설정, 사건 종료, 점검 링크 발급)
   ├─ facility.js   소방시설 점검 화면 (건물별 발급 링크로 접속해 AFP-Core/소화시설 현황을 직접 갱신)
   └─ app.js        역할 전환, 공통 상단바/진행바/SOS 버튼 제어
```

## 사건 위치는 누가 확정하는가

입주민이 시작화면에 입력하는 아파트명·동·호는 **본인이 말하는 위치(신고 접수 시 진술)**일 뿐,
그 자체가 곧바로 공식 대응 위치가 되지 않습니다. 실제 119 신고 처리 흐름을 그대로 반영했습니다.

1. 입주민이 위치를 저장하면, `js/data.js`의 `BUILDINGS` 등록 디렉터리에서 같은 아파트명·동을 자동으로
   찾아 K-apt 시설현황을 미리 채워줍니다(등록되지 않은 단지라면 직접 입력해야 합니다).
2. 입주민이 대피 안내를 끝까지 진행하면 `store.submitReport()`로 "신고"가 하나 쌓입니다. 이 시점에는
   아직 세대 현황판에 반영되지 않고, 119 상황실의 **미확정 신고** 목록에만 나타납니다.
3. **관리자 모드(`?role=admin`)가 신고를 근거로 등록 건물을 선택해 "이 위치로 화재 확정"**해야 비로소
   그 건물의 세대 그리드(Live Occupancy Status)와 AFP 정보가 채워지고, 입주민·상황실·소방대원 세 화면이
   함께 동기화되며 소방대원 화면이 활성화됩니다. 119 상황실은 이 확정 절차에 관여하지 않고, 관리자가
   확정해둔 사건들을 보며 대응하는 **읽기 전용 모니터링/대응 화면**입니다.
4. 확정된 사건의 아파트·동과 일치하지 않는 신고(다른 동 오신고, 두 번째 사건 등)는 상황실 화면에
   "다른 위치 신고 (확인 필요)"로 계속 별도 표시되어 놓치지 않도록 했습니다.

## 관리자 모드

`?role=admin` 링크로 접속하면 비밀번호(`01190119`) 입력 후 3개 화면을 총괄하는 관리자 대시보드가 열립니다.

- **화재(사건) 생성/위치 변경**: "OO아파트 O동 O호에 화재 발생"을 지정하면 즉시 입주민·119 상황실·
  소방대원 화면에 반영됩니다. 이미 확정된 사건의 위치를 다시 확인하거나 정정할 수도 있습니다.
- **동시 다발 화재 지원**: "+ 새 화재 추가"로 두 번째 이후 화재를 별도 사건으로 만들 수 있고, 사건마다
  입주민용 링크와 소방대원용 링크가 각각 독립적으로 발급됩니다.
- **배포 링크 유효시간(TTL) 관리**: 사건별로 발급되는 입주민/소방대원 링크의 만료시각을 확인하고,
  기본 유효시간을 조정하거나 "⏱ 유효시간 연장" 버튼으로 **이미 배포된 바로 그 링크**를 계속
  쓸 수 있게 늘릴 수 있습니다(링크 주소 자체는 바뀌지 않습니다 — 아래 "역할별 접근 링크" 참고).
- **사건 종료**: 상황이 종료된 사건을 닫아 세대 현황판에서 제외합니다.
- **소방시설 점검 링크 발급**: 화재 여부와 무관하게, 등록된 건물마다 `?role=facility&building=ID`
  링크를 발급할 수 있습니다. 이 링크를 해당 아파트 관리사무소에 전달하면, 정기 소방시설 점검 후
  피난시설(AFP-Core)·소화시설 설치 현황을 직접 갱신할 수 있습니다(아래 "소방시설 점검 모드" 참고).

로그인 상태는 `sessionStorage`에만 저장되는 클라이언트 측 체크로, 실제 인증이 아닙니다(아래 "알아두어야
할 한계" 참고).

## 소방시설 점검 모드

입주민 화면에는 더 이상 "시설정보가 다르면 알려주세요" 같은 수정 요청 버튼이 없습니다. 소방시설
현황은 화재 신고자가 아니라 **관리자가 건물별로 발급한 별도 링크**(`?role=facility&building=ID`,
비밀번호 없음 — `js/admin.js`의 "🧯 소방시설 점검 링크 발급" 패널에서 복사)로만 수정할 수 있습니다.

- 링크에 접속하면 그 건물의 피난시설(AFP-Core) 7종·소화시설 3종을 설치/미설치로 바로 토글해 저장할 수
  있습니다. 저장된 값은 `store.js`의 `buildingOverrides`에 건물별로 누적되어, 정적 등록 정보
  (`js/data.js`의 `BUILDINGS`)를 덮어쓰지 않고도 "현재 유효한 설치 현황"으로 취급됩니다
  (`store.getEffectiveBuilding(id)`가 항상 이 병합된 값을 돌려줍니다).
- 저장 즉시 이후에 **새로 확정되는 사건**과, 아직 사건에 매이지 않은 입주민 화면(주소만 저장한 상태)의
  시설현황 표시에 반영됩니다. 다만 **이미 확정되어 진행 중인 사건**은 확정 시점의 시설현황을 그대로
  유지합니다(활성 화재 대응 중에 근거 데이터가 갑자기 바뀌는 것을 막기 위한 의도적 설계이며, 기존의
  "세대별 AFP 예외" 기능과 같은 원칙입니다) — 점검 결과를 이미 진행 중인 사건에도 즉시 반영하려면
  상황실의 "세대별 AFP 예외" 또는 사건 재확정 절차를 함께 사용해야 합니다.
- 점검 링크는 만료시각이 없습니다(입주민/소방대원 링크와 달리 특정 화재에 종속되지 않는, 건물의
  상시 정보이기 때문입니다). 링크 자체가 유일한 접근 제어이므로, 링크 유출 시 아무나 그 건물의
  시설현황을 바꿀 수 있습니다 — 실서비스 전환 시 반드시 별도 인증이 필요합니다.

데모에는 3개 건물(행복아파트 101동/102동, 미래아파트 205동)이 미리 등록되어 있고, 서로 다른
층수·세대수·AFP 설치 현황을 갖고 있어 건물마다 판정 결과와 소방대원 화면 정보가 달라지는 것을
확인할 수 있습니다.

## 아파트 정보 일괄 등록 (평면도·소방시설현황 제외)

대구 등 실제 지역의 아파트명·동·호 목록(예: 공공데이터포털의 건축물대장정보/공동주택 단지정보)을
확보했다면, 관리자 화면의 "🏢 아파트 정보 일괄 등록 (CSV)" 패널에 `아파트명,동,호` 형식으로 한
줄씩 붙여넣어(콤마 또는 탭 구분, 엑셀에서 그대로 복사해도 됨) 등록할 수 있습니다.

- 같은 아파트명+동으로 된 행은 자동으로 하나의 동(건물)으로 묶이고, 그 동에 실제로 존재하는 호수
  목록 그대로 등록됩니다 — 기존 데모 건물처럼 "층수 × 세대/층"으로 균일하게 찍어내는 대신, 실제
  호수가 몇 층 몇 호든 그대로 반영됩니다(`store.js`의 `registerBuildings()`, `data.js`의
  `generateUnits()`가 `building.units` 명시적 목록이 있으면 그것을 그대로 씁니다).
- **평면도와 소방시설현황(피난시설·소화시설)은 이 등록에 포함되지 않습니다.** 새로 등록된 건물은
  두 항목 모두 "확인 필요" 상태로 시작하며, 위 "소방시설 점검 모드"에서 설명한 건물별 점검 링크로
  나중에 채워야 합니다. 평면도는 애초에 등록 대상이 아닙니다(공공데이터로 구하기 어렵고, 세대별이
  아니라 84A/84B 같은 타입별로 공유되는 경우가 많아 별도 체계가 필요합니다).
- 이미 등록된 아파트명+동을 다시 붙여넣으면 새로 만들지 않고 그 건물의 호수 목록만 갱신합니다(재등록
  안전).
- 등록된 건물은 `SAFEHOME.BUILDINGS`(정적 목업 디렉터리)에 그대로 병합되므로, 건물 선택 드롭다운·
  주소 자동조회(`findBuildingByAddress`) 등 기존 코드를 전혀 바꾸지 않고도 바로 인식됩니다.

## 역할별 접근 링크 (보안)

다섯 화면은 **URL로 구분된 별도 링크**로만 접근합니다. 기본 주소(`index.html`, 파라미터 없음)는
입주민 화면만 보여주고, 상황실 링크(`?role=situation`)는 상황실+소방대원 화면을, 소방대원 링크
(`?role=firefighter`)는 소방대원 화면만, 관리자 링크(`?role=admin`)는 비밀번호 입력 후 관리자
화면만, 소방시설 점검 링크(`?role=facility&building=ID`)는 그 건물의 시설 점검 화면만 보여줍니다 —
아무 링크나 열어서 여러 역할을 다 조작할 수 있던 이전 구조의 문제를 줄인 것입니다(완전한 인증은
아니며, 실서비스 전환 시 역할별 로그인이 반드시 필요합니다. `docs/expert-review.md`의 보안전문가
항목 참고).

관리자가 사건을 확정하면 화면에 **입주민용 링크**(아파트·동이 자동 지정되고 호수만 입력)와
**그 사건 전담 소방대원용 링크**(`?role=firefighter&incident=사건ID`)가 각각 생성되어, 복사해서
배포할 수 있습니다.

두 링크 모두 **기본 2시간이 지나면 접속이 차단**됩니다. 다만 만료시각을 링크(URL) 안에 직접 담지는
않습니다 — 접속 시점마다 그 링크가 가리키는 사건의 "현재" 유효시간을 store에서 조회해 판정합니다
(`app.js`의 `isLinkExpired()`). 그래서 관리자가 "⏱ 유효시간 연장" 버튼을 누르면 **이미 배포된 바로
그 링크**가 새 URL 없이 곧바로 다시 유효해집니다 — 링크 주소 자체는 절대 바뀌지 않고, 유효시간만
연장됩니다. 기본 유효시간 설정은 새로 확정되는 사건부터 적용됩니다.

## 다중 사건 지원

동시에 여러 동에서 화재가 발생해도 관리자는 사건을 각각 확정해두고, 상황실·소방대원·관리자 화면 모두
상단 탭으로 전환하며 관리합니다. 사건마다 세대 현황판·AFP 정보·환자현황이 완전히 분리되어 있습니다.
소방대원 화면도 자신이 배정된 사건(링크의 `incident` 파라미터, 또는 직접 선택)만 보므로 다른 화면이
다른 사건으로 전환해도 영향을 받지 않습니다. 사건은 관리자 모드에서 "종료" 처리할 수 있고, 종료된
사건에 매칭되어 있던 신고는 다시 미확정 상태로 돌아갑니다.

## 119 상황실 / 소방대원 화면에서 볼 수 있는 것

- **119 상황실**: 세대별 체크리스트 외에도 건물 전체의 **소방시설현황(AFP-Core)** 과 **특화구조 정보**
  (복층·다락 세대, 옥상 대피 가능 여부·최상층, 복도 형태)를 상시 패널로 볼 수 있습니다. 체크리스트의
  "피난시설 존재 여부" 항목을 클릭하면 어떤 시설이 설치되어 있는지 바로 펼쳐볼 수 있고, 소방대원에게
  전달할 메모를 남길 수 있습니다. 환자·대피 현황은 상단 타이틀 옆에 칩 형태로 요약 표시됩니다(입력은
  소방대원 화면에서만 합니다).
- **소방대원**: 피난시설(AFP-Core)과 소화시설 현황을 함께 확인할 수 있고, "실시간 세대 현황" 그리드에서
  세대를 선택하면 그 옆에 상세 정보·상태변경 버튼(구조·대피 완료/이동 유도 중/위험 재확인 — 누르면 그리드
  칸 색이 즉시 바뀝니다)·세대 개략도가 함께 표시됩니다. 119 상황실이 남긴 메모는 화면 상단에 표시됩니다.

## 초안 대비 달라진 점

기존 단일 HTML 초안(`fire_evacuation_guide_safehome_v5.html`)을 검토한 뒤, 제공해주신 **흐름도 2.0**(공동주택
화재 발생 시 행동 안내 흐름도) 기준으로 아래와 같이 다시 설계했습니다.

- **질문 세분화**: Q1(화재 위치) 6지선다, Q2(연기·화염) 5단계, Q3(현관 대피) 6지선다로 흐름도와 동일하게 맞췄습니다.
  기존 초안은 Q2/Q3가 예/아니오 이진 선택이라 실제 판단 세밀도가 떨어졌습니다.
- **대피 결과 A~G 재정의**: 흐름도의 AFP 항목(하향식 피난구·대피공간·경량칸막이·옥상대피 등)에 "완강기"가 없어,
  초안에 있던 완강기(D) 단독 분기를 없애고 흐름도와 동일한 7개 결과(A 계단대피 / B 세대내대기 / C 대피공간이동 /
  D 하향식피난구 / E 경량칸막이 / F 구조요청 / G 옥상대피)로 통일했습니다.
- **AFP 보정 로직 통합**: 초안은 먼저 결과를 정하고 나중에 `applyAfpCorrection()`으로 덧씌우는 2단계 구조였는데,
  이 과정에서 로직을 추적하기 어려웠습니다. 새 버전은 `rules.js`의 `decide()` 한 곳에서 AFP 설치 여부를 바로
  반영해 판단 근거(`notes[]`)를 함께 남기므로, 119/소방대원 화면에서 "왜 이 결론이 나왔는지" 그대로 보여줄 수 있습니다.
- **3자 화면 추가**: 기획서의 Resident / 119 Situation Room / Firefighter Protocol 개념을 반영해
  119 상황실 대시보드(신고 큐, 체크리스트, 실시간 세대 현황)와 현장 소방대원 정보카드
  (AFP-Search 기반 건물 특이사항, 구조 우선순위)를 새로 만들고, 입주민의 대피 진행 상황이
  실시간으로 두 화면에 반영되도록 공유 상태 저장소(`store.js`)를 도입했습니다.
- **다중 건물 등록 + 사건 위치 확정 워크플로우**: 처음에는 건물 정보가 통째로 하나뿐이라 입주민이 입력한
  동이 곧바로 화면에 반영됐는데, 실제로는 건물 정보가 입주민이 아니라 관리사무소/K-apt에 사전 등록되어
  있어야 하고 최초 신고 이후 정확한 위치는 119 상황실이 확인·확정해야 합니다. 이를 반영해
  `BUILDINGS` 등록 디렉터리와 상황실의 "사건 위치 확정" 절차를 추가했습니다 (자세한 내용은 위 "사건 위치는
  누가 확정하는가" 참고).
- **PWA화**: 오프라인/네트워크 불안정 상황에서도 대피 안내가 열리도록 매니페스트와 서비스워커를 추가했습니다.

## 2차 개선사항 (5인 전문가 리뷰 반영)

`docs/expert-review.md`에 입주민·119 상황실·현장 소방대원·외부 보안전문가·경연대회 심사위원·UX 디자이너
6개 관점으로 검토한 내용과, 그에 대한 실제 조치 내역이 정리되어 있습니다. 주요 반영 사항:

- **역할별 접근 링크 분리**(위 항목), **다중 사건 지원**(위 항목)
- **패닉 단축 경로**: 화염이 보이거나 현관을 열 수 없는 등 명백히 위험한 답을 고르면, 나머지 질문을
  다 마치기 전에 즉시 임시 행동요령과 119 전화 버튼부터 보여줍니다.
- **긴급 신고 알림**: 새 긴급(critical) 신고가 들어오면 상황실 화면에 소리(비프음 2회)와 화면 깜빡임으로 알립니다.
- **조치 이력**: "누가/언제/무엇을 했는지"를 기록하는 감사 로그를 추가했습니다(사건 확정, 출동 지령, 세대 상태
  변경, 대피 완료 등).
- **소방시설 현황은 소방서 등록, 입주민은 수정 요청만**: 입주민 화면의 AFP 입력 UI를 없애고 조회 전용으로
  바꿨습니다. 실제와 다르면 "수정 요청"을 남기고, 상황실 화면에서 확인·처리합니다.
- **세대(라인)별 AFP 예외**: 건물 전체가 같은 시설현황을 공유하는 게 기본이지만, 상황실이 특정 세대에만
  다른 값을 등록해 둘 수 있습니다(예: 이 라인만 경량칸막이 미설치).
- **UX**: 상황실/소방대원 화면의 참고용 패널(소방시설현황, AFP-Search, 조치 이력 등)을 접고 펼 수 있는
  아코디언으로 바꾸고, 최소 폰트 크기를 올리고, 소방대원 조작 버튼의 터치 타겟을 44px 이상으로 키우고,
  다크모드(상단바 🌙 버튼)를 추가했습니다.

## 3차 개선사항 (관리자 모드 신설 + 화면 간소화)

- **관리자 모드 신설**: 사건 생성/위치 변경, 배포 링크 발급·TTL 설정, 사건 종료 권한을 119 상황실에서
  분리해 관리자 전용 화면(`?role=admin`, 비밀번호 `01190119`)으로 옮겼습니다. 119 상황실은 이제 순수
  모니터링/대응 화면입니다 (자세한 내용은 위 "관리자 모드" 항목 참고).
- **입주민 질문 축소**: 화재위치(Q1)에서 지하주차장·공용부 옵션을 없애고, 연기/화염(Q2)의 "화염 보임"과
  "연기 많음"을 하나로 합쳤으며, 현관 대피(Q3)에서 "문손잡이 뜨겁다"/"문을 열 수 없다"를 없애고,
  복도·계단 안전 여부(Q4)를 통째로 삭제했습니다(현관 밖 상태만으로 대피 경로 안전성을 판단하도록
  `rules.js`의 판정 로직을 단순화했습니다).
- **입주민 시작화면 간소화**: 시작화면의 "긴급모드" 설명 박스와 소방시설 현황 그리드를 없애고, 상황
  질문에 곧바로 들어가도록 했습니다. 소방시설 현황은 결과 화면에서 우리 세대에 해당하는 시설만
  클릭 가능한 카드로 보여주고(샘플 데이터 `SAMPLE_ALL_INSTALLED_AFP` 사용 — 실제 대피 판정에는
  영향 없음), 클릭하면 시설별 사용 방법이 펼쳐집니다.
- **입주민 결과화면 정리**: "119 전달문" 원문 박스(복사/공유 버튼 포함)를 없애고, "📡 119로 현상황
  전송" 버튼으로 상황실·소방대원 화면에 현재 상태를 다시 알릴 수 있게 했습니다. "가족 공유"는
  독립 버튼으로 남겨두었습니다.
- **소방대원 화면 재구성**: 층별로 나열되던 "구조 우선순위" 카드 목록을 없애고, "실시간 세대 현황"
  그리드를 확대해 그 자체가 조작 인터페이스가 되도록 했습니다 — 세대를 선택하면 옆에 상세 정보,
  상태변경 버튼(누르면 그리드 칸 색도 함께 바뀜), 세대 개략도가 표시됩니다. 피난시설(AFP-Core) 현황도
  소화시설과 나란히 볼 수 있게 추가했습니다.
- **119 상황실 정리**: "출동 지령 내리기" 버튼을 없앴고, 환자·대피 현황을 별도 아코디언 패널 대신
  타이틀 옆 요약 칩으로 옮겼습니다.

## 4차 개선사항 (소방시설 점검 모드 분리)

입주민 시작화면의 "시설정보가 다르면 알려주세요"는 119 상황실로 텍스트 요청만 보내는 기능이라
실제로 시설현황을 바로잡을 방법이 없었습니다. 이를 없애고, 관리자가 건물별로 발급하는 별도
점검 링크(`?role=facility&building=ID`)에서 그 아파트가 직접 설치 현황을 갱신하도록 바꿨습니다
(자세한 내용은 위 "소방시설 점검 모드" 참고). `store.js`에 `buildingOverrides`/`getEffectiveBuilding()`을
추가해, 등록 건물의 정적 목업 값 위에 점검으로 갱신된 값을 덮어써서 보여주도록 했습니다.

## 알아두어야 할 한계 (데모 범위)

- **건물 디렉터리(`js/data.js`의 `BUILDINGS`)는 목업 3개 동뿐**입니다. 실제 서비스에서는 K-apt 단지코드 API 및
  소방청 AFP 데이터베이스에서 전국 단지를 조회하도록 `findBuildingByAddress()`를 교체해야 합니다.
- **119 상황실 → 출동대 지령, 소방대원 처치 결과**는 실제 무선지령시스템이 아니라 화면상의 상태 표시(mock)입니다.
- **실시간 동기화는 같은 브라우저(localStorage 기준)** 로 제한됩니다. 여러 사용자가 서로 다른 기기에서 함께
  보려면 Firebase Realtime Database, Supabase 등 실제 백엔드로 `store.js`의 저장/구독 부분만 교체하면 됩니다.
- **역할별 링크 분리는 완전한 인증이 아닙니다.** 링크(URL)만으로 화면을 구분할 뿐, 로그인이나 서버 검증이
  없어서 링크 URL을 아는 사람은 누구나 개발자도구 콘솔에서 `SAFEHOME.store`를 직접 호출해 상태를 조작할
  수 있습니다. 실서비스 전환 시 역할별 로그인/서버 측 권한 검증이 반드시 필요합니다.
- **관리자 모드 비밀번호도 실제 보안이 아닙니다.** 클라이언트 JS에 하드코딩된 문자열 비교(`sessionStorage`에
  로그인 여부만 저장)일 뿐이라, 개발자도구로 `js/admin.js`를 열어보면 비밀번호가 그대로 노출됩니다.
  실서비스 전환 시 서버 측 인증(세션/토큰 검증)으로 반드시 교체해야 합니다.
- **소방시설 점검 링크는 비밀번호도 만료시각도 없습니다.** 링크 자체가 유일한 접근 제어라, URL이
  유출되면 누구나 그 건물의 시설현황을 바꿀 수 있습니다. 또한 이미 진행 중인 사건은 점검으로 갱신된
  값을 자동으로 반영하지 않습니다(의도된 설계이며, 위 "소방시설 점검 모드" 항목 참고).
