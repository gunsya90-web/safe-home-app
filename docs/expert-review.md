# SAFE-HOME 5(+1)인 전문가 관점 분석

작성일: 2026-07-04
대상: `safehome-app` 현재 코드 기준 (커밋 `9697ec3` 시점)

입주민 · 119 상황실 · 현장 소방대원 · 외부 보안전문가 · 경연대회 심사위원 5개 관점에
UX/UI 디자이너 관점을 더해 실제 코드(escaping 처리, 인증 여부, 반응형 레이아웃 등)를
확인하며 정리했다.

## 1. 입주민 (실제 화재 상황의 최종 사용자)

### 문제점
- **주소 입력이 긴급 상황 앞에 걸려있음.** 지금 흐름은 시작화면에서 "우리집 정보 저장 →
  K-apt 조회/입력 → 긴급 안내 시작"을 거친다. 실제 화재 중 이 절차를 처음부터 밟는 건
  비현실적이다. 평상시(비긴급) 등록과 긴급 시 진입을 분리해야 한다.
- **한국어 전용.** 대구에도 외국인 거주자가 있을 수 있는데 다국어 지원이 전혀 없다.
  생명안전 앱에서는 우선순위가 높은 항목이다.
- **패닉 상태를 위한 배려 부족.** 5개 질문을 순서대로 다 거쳐야 결과가 나온다. 이미
  명백히 위험한 상황(예: 화염이 보임)이면 즉시 "지금 당장 해야 할 것"부터 보여주고,
  나머지 질문은 선택적으로 진행하는 식의 단축 경로가 필요하다.
- **음성 안내가 결과 화면에만 있음.** 질문을 읽어주는 기능이 없어서, 시각장애인·저시력
  고령자는 질문 단계에서부터 불리하다.
- 대피 완료 버튼을 누르지 않으면 상황실은 그 세대가 계속 "위험/이동중" 상태로 인지한다
  — 대피 후 앱을 놓고 뛰쳐나가는 게 현실적인데, 이 경우 갱신이 안 된다.

### 추가 필요사항
- 평상시 온보딩(주소 등록) / 긴급모드 진입을 분리
- 질문 단계에도 TTS 적용, 큰 글씨·고대비 모드
- "지금 매우 위험함" 원터치 SOS 경로(질문 스킵하고 바로 구조요청+위치전송)
- 다국어(최소 영어) 지원

## 2. 119 상황실

### 문제점
- **동시에 사건 1건만 처리 가능.** `state.incident`가 단일 객체라, 실제로 여러 동네에서
  동시에 화재가 나면 지금 구조로는 처리가 안 된다. 데모를 넘어서는 순간 가장 먼저
  부딪힐 구조적 한계다.
- **새 위험 신고에 대한 알림이 없음.** 배지 숫자만 바뀔 뿐, 소리·깜빡임 같은 능동적
  알림이 없어서 다른 일 하던 상황실 요원이 놓칠 수 있다.
- **누가 언제 무엇을 했는지 기록이 없음.** "이 세대를 안전으로 바꾼 사람이 누구였는지"
  같은 감사(audit) 로그가 전혀 없다. 사후 검토·책임소재 파악에 필요하다.
- 건물의 AFP 정보가 동 전체에 균일하게 적용된다. 실제로는 같은 동이라도 라인·세대별로
  시설이 다를 수 있는데 지금은 반영이 안 된다.

### 추가 필요사항
- 다중 사건 지원(사건 리스트 + 사건 전환)
- 신규/긴급 신고 사운드 알림
- 조치 이력(누가/언제/무엇) 로그
- 세대(라인)별 AFP 편차 반영 가능한 구조

## 3. 현장 소방대원

### 문제점
- **현장 연결성 가정이 비현실적.** 건물 안으로 진입하면 통신이 약해지는 게 정상인데,
  지금은 "오프라인이면 예전에 캐시된 화면"이 뜰 뿐 "이 데이터가 최신이 아닐 수 있다"는
  경고가 없다. 화재 진압 중 세대 상태가 갱신됐는데 그걸 모르고 판단하면 위험하다.
- **터치 타겟이 작음.** 구조 우선순위 카드의 버튼(구조·대피 완료 / 이동 유도 중 / 위험
  재확인)이 3분할이라 글자·버튼이 작다. 방화장갑 낀 손으로 조작한다고 가정하면 너무
  정밀한 터치를 요구한다.
- 무전/실제 지휘체계와 완전히 분리된 별도 앱이라, 현실에서는 "손에 폰 들고 탭하기"가
  진압 중엔 어려울 수 있다 — 데모의 한계로 인지하고 넘어가되, 심사에서 나올 수 있는
  질문이다.

### 추가 필요사항
- "마지막 갱신 시각" 표시 + 오프라인/데이터 지연 경고 배너
- 버튼 크기 확대(최소 44px 터치 타겟 권장), 3분할 대신 2분할이나 세로 배치 고려
- (장기) 음성 명령/간단 제스처로도 상태 갱신 가능하게

## 4. 외부 보안전문가

### 문제점 (심각도순)

| 항목 | 내용 | 심각도 |
|---|---|---|
| 인증/권한 없음 | URL만 알면 누구나 "119 상황실"·"소방대원" 탭을 눌러 진짜처럼 조작 가능. 허위로 세대를 "안전"으로 표시하거나 가짜 출동지령을 내릴 수 있음 | **높음** |
| 클라이언트 단독 신뢰 | 모든 "확정된" 데이터가 브라우저 localStorage에만 있어서, 개발자도구 콘솔에서 누구나 `SAFEHOME.store`를 직접 호출해 상태를 조작 가능 | **높음** |
| 초기화 버튼 무방비 | 상단 초기화 버튼이 confirm() 팝업 하나로 전체 데이터(진행 중인 실제 상황 포함)를 삭제 가능 | 중간 |
| 평문 저장 | localStorage에 위치·환자현황 등이 암호화 없이 저장됨 | 중간 |
| 스팸/도배 방지 없음 | `submitReport`에 횟수 제한이 없어 허위 신고를 무한정 등록 가능 | 중간 |
| CSP 미설정 | innerHTML을 광범위하게 쓰지만(이스케이프 처리는 잘 되어 있음, 확인함) CSP 헤더/메타태그가 없어 이중 방어선이 없음 | 낮음 |

**잘 되어 있는 점**: 사용자 입력값(주소, 메모 등)이 렌더링되는 모든 지점에서
`escapeHtml()`이 일관되게 적용되어 있어 기본적인 XSS는 잘 막혀 있다.

### 추가 필요사항
- 역할별 로그인/PIN 최소한이라도 도입 (특히 상황실·소방대원 탭)
- 실서비스 전환 시 서버 검증 계층 필수 (지금은 "신뢰의 경계"가 사실상 없음)
- 초기화 버튼에 2단계 확인 또는 접근 제한
- Naver/K-apt API 키 연동 시, 공개 저장소에 시크릿 키가 절대 들어가지 않도록
  `.env` + `.gitignore` 체계를 지금 미리 준비

## 5. UX/UI 디자이너

전반적으로 3개 화면(입주민 모바일 카드형 / 상황실·소방대원 데스크톱 대시보드형)을
역할에 맞게 다르게 설계한 건 좋은 판단이다. 다만 세부적으로:

- **정보 밀도 문제**: 상황실·소방대원 화면은 패널이 세로로 계속 쌓이는 구조라(소방시설현황
  → 특화구조 → 환자현황 → 소화시설 → AFP-Search → 신고 큐 → 세대현황 → …) 스크롤이
  매우 길다. 실제 위급 상황에서 "지금 당장 봐야 할 것"이 한눈에 안 들어올 수 있다.
  탭/아코디언으로 섹션을 접고 펼 수 있게 하거나, 우선순위 높은 패널(신고 큐, 구조
  우선순위)을 상단 고정하는 걸 권한다.
- **색상 신호 체계는 일관적이고 좋음**: 위험(빨강)/이동중(주황)/대기(파랑)/완료(초록)/
  미응답(회색) 색 규칙이 전 화면에서 동일하게 쓰여서 학습 부담이 적다.
- **작은 글씨가 많음**: 11~12px 텍스트가 상당히 많다(체크리스트, 소화시설 그리드 등).
  데스크톱 모니터 기준으로는 괜찮지만, 소방대원이 실제로는 태블릿/폰으로 볼 가능성이
  높다는 걸 고려하면 최소 폰트 크기를 좀 더 키우는 게 안전하다.
- **다크모드 없음**: 화재 현장은 야간·정전 상황이 흔한데, 지금은 밝은 배경 고정이라
  야간 눈부심·배터리 소모 측면에서 불리하다.
- **로딩/오류 상태 부재**: 지금은 모든 게 즉시 렌더링되지만(로컬 데이터라), 실제 API
  연동 후에는 "불러오는 중", "실패했습니다" 같은 상태 표현이 전혀 준비되어 있지 않다.
- **세대 도면·체크리스트 펼치기 등 인터랙션은 직관적**: 클릭하면 바로 펼쳐지는 패턴이
  일관돼 있어 배우기 쉽다.

### 추가 필요사항
- 상황실/소방대원 화면에 섹션 접기/펼치기 또는 탭 구조 도입
- 최소 폰트 크기 상향(특히 소방대원 화면)
- 다크모드
- API 연동 대비 로딩/오류 상태 디자인 미리 준비

## 6. 경연대회 심사위원

### 강점으로 보일 부분
- 기획서의 "3자(입주민·119·소방대원) 동시 활용" 컨셉을 실제로 동작하는 데모로 구현한
  점 — 말로만 하는 기획서가 아니라 클릭해서 확인 가능
- "신고 = 주장, 상황실 확정 = 공식화"라는 현실적인 워크플로우를 반영한 것 — 단순
  챗봇을 넘어선 시스템 설계로 보임
- 자동감지 신호로 시작하는 데모 시나리오가 미리 세팅되어 있어 발표 시연이 매끄러움

### 심사위원이 물어볼 만한 질문 (미리 답 준비 필요)
- "실제 AFP/K-apt 데이터와 연동됩니까?" → 지금은 목업. 연동 로드맵은 README에 정리됨
- "여러 사건 동시 처리 됩니까?" → 안 됨(위 상황실 항목 참고), 확장 계획 필요
- "정량적 효과(골든타임 단축 등)를 보여줄 수 있습니까?" → 지금은 없음. 시뮬레이션
  기반 수치라도 하나 준비하면 설득력이 커짐
- "보안/인증은요?" → 프로토타입 단계임을 명확히 인정하고, 실서비스 로드맵에서 다룰
  항목으로 제시하는 게 방어적으로 안전

### 추가 필요사항
- 발표용 "Before/After" 또는 "골든타임 단축 시뮬레이션" 수치 1개
- 로드맵 슬라이드(현재 프로토타입 → 실증 → 실서비스 단계) 명확화

## 우선순위 정리 (지금 손대면 효과 큰 순서)

1. **입주민 긴급모드/평상시모드 분리** — 실사용 현실성에 가장 직결
2. **역할별 최소 인증(PIN)** — 보안전문가 지적 중 가장 심각하고, 구현 부담은 상대적으로 작음
3. **상황실/소방대원 화면 섹션 접기** — 정보과다 문제 해결, 심사 시연 때도 더 깔끔해 보임
4. **다중 사건 지원** — 구조적으로 크지만, 심사위원 질문에 대비해 최소한 "로드맵으로
   인지하고 있다"는 답이라도 필요
